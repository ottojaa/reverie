## Entries

- `2026-07-19` — **`thumbnail_status == COMPLETE` does NOT mean a thumbnail image exists.** The backend marks non-previewable files (apk, zip, audio, exe — anything whose `getThumbnailStrategy` is `none`) `thumbnail_status: 'complete'` at upload with no image ever rendered (`upload.service.ts`: `canThumbnail ? 'pending' : 'complete'`). The real "has a rendered thumbnail" signal is `thumbnail_paths != null` (what the `/documents/:id/thumbnail/:size` route gates its 404 on, and what the thumbnail worker writes alongside `thumbnail_blurhash`). Web's FallbackViewer sidesteps this by always showing a file-type icon. Android can't use `thumbnail_paths` because the Room cache drops it (`Mappers.toDto` sets it null) and both grid + viewer read through the cache — so the cache-safe proxy is **`thumbnail_blurhash` presence** (`DocumentDto.hasRenderedThumbnail` in `DocumentDtos.kt`). Bug this fixed: APKs rendered an empty black "preview" in the viewer and a blank grid tile because status==complete drove an AsyncImage the server 404'd. Used by `FallbackViewer`, `DocumentThumbnail`, `ViewerRegistry`.
- `2026-07-18` — **shadcn Button ghost/outline hover is indigo (`accent`) in this app** — the repo maps `--accent` to indigo (not the usual neutral), and Button's ghost variant carries `dark:hover:bg-accent/50`. Restyling a ghost Button's hover with a plain `hover:bg-*` class does NOT win in dark mode: twMerge only dedupes within the same variant stack, so `dark:hover:bg-accent/50` survives and the control flashes indigo. Any custom-styled ghost/outline Button (filter pills, panel rows, segmented controls) needs an explicit `dark:hover:bg-…` override alongside the `hover:bg-…` one. See `pillStateClass` in `apps/web/src/components/search/filter-bar/FilterPill.tsx`.
- `2026-07-18` — **Search q-DSL grammar is defined once in `libs/shared/src/search/query-tokens.ts`** (tokenizeQuery/serializeQuery + add/remove/replace/setFreeText helpers). Backend `query-parser.ts` consumes it for tokenization (semantics stay backend-side); ALL web filter UI mutates `q` through it (`useSearchQuery` hook, `FilterMenuButton`). Never regex-parse or string-replace the `q` param again — that's the bug class this replaced (three drifting regex parsers + substring removals). Backend behavior is pinned by a 40-case parity corpus in `apps/backend/src/search/query-parser.test.ts`.

- `2026-07-13` — Package manager is **Yarn 4 (Berry)** (migrated off pnpm, then off Yarn Classic). Pinned in-repo at `.yarn/releases/yarn-4.17.1.cjs` + `.yarnrc.yml` (`nodeLinker: node-modules`), so **corepack is never needed** — in Docker/CI invoke it as `node .yarn/releases/yarn-4.17.1.cjs <cmd>`. Internal deps use `workspace:*`. `resolutions` pins the `nx`/`@nx/*` ecosystem to 22.4.0 (mismatched plugin minors break the project graph with a `Target.inputs` error). `.yarnrc.yml` sets `npmMinimalAgeGate: 14d` — a supply-chain cooldown that refuses package versions younger than 14 days (Yarn auto-picks the newest older release per range; only exact transitive pins that are too new need a resolution or a loosened range, e.g. `@anthropic-ai/sdk` is `^0.107.0` not `^0.111.0`). If a fresh `yarn install` errors with "All versions satisfying X are quarantined", that's the age gate, not npm.
- `2026-07-13` — Docker prod install uses Berry's `yarn workspaces focus @reverie/backend --production` in the builder (Debian `node:22`, for native-module ABI parity with the `node:22-slim` runtime), then copies the focused `node_modules` + `apps/backend/dist` + `libs/shared` into the runtime (no install there). `@reverie/shared` is a symlink in `node_modules` → `libs/shared`, so `libs/shared` (built) must be present at that relative path. Local Docker validation on Apple Silicon fails only at the PaddleOCR pip step (paddlepaddle wheels are x86_64-only); the prod host is x86_64 so that's fine.
- `2026-07-14` — Do **not** add a separate `vitest.config.ts` next to `apps/web/vite.config.mts`: the `@nx/vite` plugin picks it up as _the_ config for inferred targets and the **build** then fails with "Could not resolve entry module index.html". Put the `test: {}` block inside `vite.config.mts` instead (web unit tests live in `src/**/*.test.ts`, run via `nx run @reverie/web:test`).
- `2026-07-14` — **Yarn Berry bin visibility**: a workspace's package.json scripts only see binaries of deps declared in _that workspace_ — root devDependencies' bins (`concurrently`, `vite`, `electron`, `tsc`, `wait-on`…) are NOT inherited like npm/Yarn 1 PATH-walking. Symptom: `command not found: concurrently` from `yarn dev:electron`. Fix: declare the tools a workspace's scripts invoke in that workspace's own devDependencies (mirror the root version ranges so the lockfile resolution — and the 14-day age gate — are unaffected).
- `2026-07-14` — **Backend esbuild output is nested**: `@nx/esbuild` with `bundle: false` emits to `apps/backend/dist/apps/backend/src/...` (plus a `dist/main.js` shim), so `__dirname`-relative paths that count directory levels differ between tsx runs (from `src/`) and `nx serve` (from `dist/`). `config/env.ts` now walks upward (`findUp`) to locate the repo-root `.env`. Also: the `require('dotenv/config')` in `main.ts` cannot rescue env loading — esbuild hoists the ESM imports (including `env.ts`, which validates and `process.exit(1)`s) above it.
- `2026-07-14` — Canvas mode (`apps/web/src/components/canvas/`) bundle boundary: **only files under `components/canvas/scene/` may import `three`/`@react-three/*`** — `CanvasScene.tsx` is the `React.lazy` entry, so three.js (~1 MB chunk) stays out of the main bundle. Any import of `scene/*` from shared code drags three into `index-*.js`; verify with `grep -l WebGLRenderer apps/web/dist/assets/index-*.js` after build (must be empty). `canvas/dive/` and `canvas/layout/` are deliberately three-free so `CanvasPage` (main chunk) can import them.
- `2026-07-14` — Canvas scene color management: custom ShaderMaterials (ground grid, document cards) skip three's colorspace output encode, so **all color/texture inputs stay raw sRGB** — uniforms via `setRawColor()` (`scene/glColor.ts`), textures left in `NoColorSpace`. Mixing in a converted color (plain `new Color(hex)`) will render visibly wrong (washed out/dark). Theme colors are oklch CSS vars, resolved to hex via a 1×1 2D canvas in `scene/theme.ts` (THREE.Color can't parse oklch). Also: `flipY` is ignored for typed-array uploads (blurhash `DataTexture`) — rows are flipped manually in `textureCache.ts`.
- `2026-07-14` — **Never run two Vite dev servers for apps/web at once**: they share one dep cache (`cacheDir: node_modules/.vite/apps/web`), and whichever re-optimizes (e.g. on a yarn.lock change) clobbers the other's chunks → the older server permanently serves `504 Outdated Optimize Dep` for every dep (browser _and_ Electron — it's server-side) until restarted. Guardrails: `dev:electron` runs `vite --strictPort` + `concurrently -k`, and `optimizeDeps.include: ['lucide-react/dynamic']` pre-bundles the ~1600 per-icon chunks (`ban-*.js`, `ear-*.js`… = lucide icon names) at startup so first-open of an icon modal can't trigger a mid-session re-optimization. Remedy when it happens: kill all vite/nx-serve instances, `rm -rf node_modules/.vite/apps/web`, start one server.
- `2026-07-14` — **Electron entry is ESM, preload is CJS** (`apps/web/electron/`): `main.mts` → `main.mjs` (package.json `"main"`), `preload.cts` → `preload.cjs` — sandboxed preloads can't be ESM. `electron/tsconfig.json` uses `module: nodenext` so the emitted format is extension-driven; don't rename these back to `.ts` (apps/web is `"type": "module"`, so a CJS-compiled `.js` crashes with "exports is not defined"). `dev:electron` runs vite with `--strictPort`: Electron hardcodes `localhost:4200`, so a busy 4200 (e.g. `nx serve web` in another tab) must fail fast instead of Electron silently attaching to the other server.
- `2026-07-16` — **R3F pointer-capture must be released explicitly**: R3F's `capturedMap` cleanup after implicit (pointerup) capture release happens in a `requestAnimationFrame` callback; until it runs, every raycast force-includes the captured object, which suppresses `onPointerMissed` (canvas click-away). In throttled/hidden tabs that rAF never fires → click-away dead until reload. Pattern: any R3F `setPointerCapture(e.pointerId)` needs a matching `releasePointerCapture` in pointerup (see `FolderIsland.tsx`).

- `2026-07-17` — **Structured outputs require the `zod/v4` subpath**, not the default `zod` entry. `@anthropic-ai/sdk` 0.107's `helpers/zod` does `import * as z from 'zod/v4'`, so a schema built with the repo-default `import { z } from 'zod'` (v3 API, shipped alongside v4 in zod 3.25.76) is a type-mismatch against `zodOutputFormat` and makes `.parse()` infer `parsed_output: {}`. Fix: build any schema passed to `zodOutputFormat` with `import * as z from 'zod/v4'` (see `apps/backend/src/llm/llm-response.schema.ts`). Keep such schemas backend-local — they're model contracts, not the v3 `@reverie/shared` API contracts.
- `2026-07-17` — **LLM name hallucinations leak into free-text, not just entities.** Diagnosed on prod doc `e32b273f...`: PaddleOCR read "TIETOENATOR" correctly (97% conf), entity grounding kept it correct, yet the model still wrote "Tietonenator" in the _summary_ and a _tag_. Lesson: OCR fine-tuning is the wrong lever when OCR is already right — the leak is the model re-typing proper nouns from memory in ungrounded free-text. Fix: `spelling-corrector.ts` scrubs summary/title/topics/tags against the _grounded_ entities (edit-distance, not confusion classes — a letter insertion isn't a visual confusion). Conservative: only rewrites a word ≥6 chars that is a close, unambiguous variant of a proper noun actually present in the OCR text and not itself a document token. Wired in `llm.service.ts` after `groundEntities`.
- `2026-07-17` — **OCR errors on _good_ images were a model-generation gap, fixed by PP-OCRv5→PP-OCRv6.** On a clean 97%-confidence Finnish receipt, `PP-OCRv5_server` systematically dropped umlauts (Määrä→Mäarä, SÄILYTTÄKÄÄ→SAILYTTAKAA, JAAKONMÄKI→JAAKONMAKI) and mangled small print (footer `…tililt…yhtensEUR 2.51882`). `PP-OCRv6_medium` (installed already in PaddleOCR 3.7 / PaddleX 3.7.2 — no upgrade, no age-gate) reads all of it correctly (99% conf) and even recovers fine print v5 missed. Change is just the model names in `ocr_runner._build_ocr` (+ ENGINE_NAME). Tested empirically with `apps/backend/ocr_service/ocr_compare.py` (config-matrix harness; needs GPU with the backend stopped — the live service reserves ~92% of the RTX 3080 via Paddle's default `fraction_of_gpu_memory_to_use`, so a co-resident run OOMs; CPU OOMs on 4k images). Two things NOT to change: Node-side preprocessing (grayscale+normalize+sharpen) is ~neutral, not harmful; and turning the doc-preprocessor OFF or tuning detection (limit_side_len/unclip) made recognition much worse. Deploy: **no Dockerfile change needed**, and do NOT edit `Dockerfile.ocr-base` — it's in `deploy.sh`'s `OCR_BASE_INPUTS` hash, so any edit (even a comment) triggers a needless multi-GB CUDA base rebuild. `ocr_runner.py` is app code baked by the normal per-deploy `compose build backend`; v6 models download at runtime into the persisted `ocr_models` volume (`/root/.paddlex`, compose) and self-heal on stall, so only the first OCR after deploy pays the download. (Corrects the stale "no model-cache volume" claim — the volume fix is in the repo now.)

- `2026-07-18` — **Android redacts GPS EXIF at pick time — web uploads from phones can never carry location.** Since Android 10, photos read without `ACCESS_MEDIA_LOCATION` get location EXIF redacted to empty structures (`GPSLatitude: [null,null,null]`, `GPSTimeStamp: "NaN:NaN:NaN"`), and the system photo picker (used by Chrome file inputs and the Files app) always redacts. Nothing web-side can bypass this. When the Android app's upload flow gets built (`apps/android` — currently a stub with no picker), it must request `ACCESS_MEDIA_LOCATION` and read picked photos via `MediaStore.setRequireOriginal(uri)` (NOT the Photo Picker, which redacts unconditionally) to preserve GPS. Desktop uploads of cable-transferred files keep GPS intact — that's why the February 2026 batch has location and phone-browser uploads don't.
- `2026-07-18` — **Documents may share one storage object — file deletion MUST refcount.** Upload hash-dedup was removed (same content + any name = separate document rows), but storage stays content-addressed, so N documents can point at one `file_path`. Any code deleting a document's file must go through `uploadService.deleteDocumentFiles()` (refcounts `file_path` across documents before removing the object; thumbnails are per-document `thumbnails/<docId>/…` and always safe to delete). Raw `storageService.deleteFile(doc.file_path)` on a document is a data-loss bug. Exception: whole-account wipes (user.service) may delete unconditionally.
- `2026-07-18` — **EXIF GPS NaN trap.** exifr's pick-config computes `latitude: NaN` from empty GPS rationals (`GPSLatitude: [null,null,null]` — what the Android photo picker's location redaction leaves behind, see the Android-redaction entry; Google Drive/Photos themselves preserve EXIF), and `typeof NaN === 'number'` / `NaN != null` let it through both the extractor check and the worker's insert guard into `photo_metadata` (Postgres `real` stores NaN). Downstream, `z.number()` rejects NaN → the document detail response 500s. Guards now: `Number.isFinite` in `ocr/exif-extractor.ts` (source) and NaN→null in `documents.route.ts toDocumentPhotoMetadata` (legacy rows). Capture date (`DateTimeOriginal`) survives the picker redaction; GPS does not. Location-preserving ingest path: Google Photos auto-backup (native app, holds media-location access) → download own photos on desktop → upload from desktop.

- `2026-07-17` — Per-document LLM analysis (`apps/backend/src/llm/`) hardening: (1) `summarizeDocument` uses `messages.parse` + `output_config.format` (structured output) + `temperature: 0`, validated against `LlmAnalysisSchema`; the model now returns an explicit `tags` array. (6) Fixed dead search wiring: entity filter/facet now read `llm_results.metadata->'entities'` (was the never-written `keyEntities` / always-null `ocr.metadata->'companies'`).
