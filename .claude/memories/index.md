## Entries

- `2026-07-13` ‚Äî LLM provider is **Anthropic (Claude)**, not OpenAI. Single integration seam: `apps/backend/src/llm/anthropic.client.ts` (all LLM calls go through it). Models are env-configured: organize ‚Üí `ANTHROPIC_ORGANIZE_MODEL` (Claude Sonnet 5, agentic tool-calling); per-document summary + vision ‚Üí `ANTHROPIC_SUMMARY_MODEL`/`ANTHROPIC_VISION_MODEL` (Claude Haiku 4.5, high-volume/cheap). `ANTHROPIC_EFFORT` sets organize thinking depth.
- `2026-07-13` ‚Äî Organize (`services/organize.service.ts`) uses the Anthropic Messages API, which is **stateless** (no OpenAI `previous_response_id`). Multi-turn state (message history + `group_id` ‚Üí document-id stash) lives in Redis via `services/organize-conversation.store.ts`, keyed by the opaque `response_id` the web client round-trips (TTL 1h). The system prompt + tool defs are cached via `cache_control`.
- `2026-07-13` ‚Äî Organize perf: `find_documents` never returns raw document UUIDs to the model ‚Äî it hands back short `group_id`s and stashes the ids server-side; `propose_organization` references groups by `group_id`. Keeps context/token cost small on the stateless API. Don't reintroduce UUID lists into tool output.
- `2026-07-13` ‚Äî Package manager is **Yarn 4 (Berry)** (migrated off pnpm, then off Yarn Classic). Pinned in-repo at `.yarn/releases/yarn-4.17.1.cjs` + `.yarnrc.yml` (`nodeLinker: node-modules`), so **corepack is never needed** ‚Äî in Docker/CI invoke it as `node .yarn/releases/yarn-4.17.1.cjs <cmd>`. Internal deps use `workspace:*`. `resolutions` pins the `nx`/`@nx/*` ecosystem to 22.4.0 (mismatched plugin minors break the project graph with a `Target.inputs` error). `.yarnrc.yml` sets `npmMinimalAgeGate: 14d` ‚Äî a supply-chain cooldown that refuses package versions younger than 14 days (Yarn auto-picks the newest older release per range; only exact transitive pins that are too new need a resolution or a loosened range, e.g. `@anthropic-ai/sdk` is `^0.107.0` not `^0.111.0`). If a fresh `yarn install` errors with "All versions satisfying X are quarantined", that's the age gate, not npm.
- `2026-07-13` ‚Äî Docker prod install uses Berry's `yarn workspaces focus @reverie/backend --production` in the builder (Debian `node:22`, for native-module ABI parity with the `node:22-slim` runtime), then copies the focused `node_modules` + `apps/backend/dist` + `libs/shared` into the runtime (no install there). `@reverie/shared` is a symlink in `node_modules` ‚Üí `libs/shared`, so `libs/shared` (built) must be present at that relative path. Local Docker validation on Apple Silicon fails only at the PaddleOCR pip step (paddlepaddle wheels are x86_64-only); the prod host is x86_64 so that's fine.
- `2026-07-14` ‚Äî Do **not** add a separate `vitest.config.ts` next to `apps/web/vite.config.mts`: the `@nx/vite` plugin picks it up as _the_ config for inferred targets and the **build** then fails with "Could not resolve entry module index.html". Put the `test: {}` block inside `vite.config.mts` instead (web unit tests live in `src/**/*.test.ts`, run via `nx run @reverie/web:test`).
- `2026-07-14` ‚Äî **Yarn Berry bin visibility**: a workspace's package.json scripts only see binaries of deps declared in _that workspace_ ‚Äî root devDependencies' bins (`concurrently`, `vite`, `electron`, `tsc`, `wait-on`‚Ä¶) are NOT inherited like npm/Yarn 1 PATH-walking. Symptom: `command not found: concurrently` from `yarn dev:electron`. Fix: declare the tools a workspace's scripts invoke in that workspace's own devDependencies (mirror the root version ranges so the lockfile resolution ‚Äî and the 14-day age gate ‚Äî are unaffected).
- `2026-07-14` ‚Äî **Backend esbuild output is nested**: `@nx/esbuild` with `bundle: false` emits to `apps/backend/dist/apps/backend/src/...` (plus a `dist/main.js` shim), so `__dirname`-relative paths that count directory levels differ between tsx runs (from `src/`) and `nx serve` (from `dist/`). `config/env.ts` now walks upward (`findUp`) to locate the repo-root `.env`. Also: the `require('dotenv/config')` in `main.ts` cannot rescue env loading ‚Äî esbuild hoists the ESM imports (including `env.ts`, which validates and `process.exit(1)`s) above it.
- `2026-07-14` ‚Äî Canvas mode (`apps/web/src/components/canvas/`) bundle boundary: **only files under `components/canvas/scene/` may import `three`/`@react-three/*`** ‚Äî `CanvasScene.tsx` is the `React.lazy` entry, so three.js (~1 MB chunk) stays out of the main bundle. Any import of `scene/*` from shared code drags three into `index-*.js`; verify with `grep -l WebGLRenderer apps/web/dist/assets/index-*.js` after build (must be empty). `canvas/dive/` and `canvas/layout/` are deliberately three-free so `CanvasPage` (main chunk) can import them.
- `2026-07-14` ‚Äî Canvas scene color management: custom ShaderMaterials (ground grid, document cards) skip three's colorspace output encode, so **all color/texture inputs stay raw sRGB** ‚Äî uniforms via `setRawColor()` (`scene/glColor.ts`), textures left in `NoColorSpace`. Mixing in a converted color (plain `new Color(hex)`) will render visibly wrong (washed out/dark). Theme colors are oklch CSS vars, resolved to hex via a 1√ó1 2D canvas in `scene/theme.ts` (THREE.Color can't parse oklch). Also: `flipY` is ignored for typed-array uploads (blurhash `DataTexture`) ‚Äî rows are flipped manually in `textureCache.ts`.
- `2026-07-14` ‚Äî **Never run two Vite dev servers for apps/web at once**: they share one dep cache (`cacheDir: node_modules/.vite/apps/web`), and whichever re-optimizes (e.g. on a yarn.lock change) clobbers the other's chunks ‚Üí the older server permanently serves `504 Outdated Optimize Dep` for every dep (browser _and_ Electron ‚Äî it's server-side) until restarted. Guardrails: `dev:electron` runs `vite --strictPort` + `concurrently -k`, and `optimizeDeps.include: ['lucide-react/dynamic']` pre-bundles the ~1600 per-icon chunks (`ban-*.js`, `ear-*.js`‚Ä¶ = lucide icon names) at startup so first-open of an icon modal can't trigger a mid-session re-optimization. Remedy when it happens: kill all vite/nx-serve instances, `rm -rf node_modules/.vite/apps/web`, start one server.
- `2026-07-14` ‚Äî **Electron entry is ESM, preload is CJS** (`apps/web/electron/`): `main.mts` ‚Üí `main.mjs` (package.json `"main"`), `preload.cts` ‚Üí `preload.cjs` ‚Äî sandboxed preloads can't be ESM. `electron/tsconfig.json` uses `module: nodenext` so the emitted format is extension-driven; don't rename these back to `.ts` (apps/web is `"type": "module"`, so a CJS-compiled `.js` crashes with "exports is not defined"). `dev:electron` runs vite with `--strictPort`: Electron hardcodes `localhost:4200`, so a busy 4200 (e.g. `nx serve web` in another tab) must fail fast instead of Electron silently attaching to the other server.
- `2026-07-16` ‚Äî **R3F pointer-capture must be released explicitly**: R3F's `capturedMap` cleanup after implicit (pointerup) capture release happens in a `requestAnimationFrame` callback; until it runs, every raycast force-includes the captured object, which suppresses `onPointerMissed` (canvas click-away). In throttled/hidden tabs that rAF never fires ‚Üí click-away dead until reload. Pattern: any R3F `setPointerCapture(e.pointerId)` needs a matching `releasePointerCapture` in pointerup (see `FolderIsland.tsx`).
- `2026-07-16` ‚Äî Canvas unravel is **explicit-open only**: folders open via `unravelRequest` (island click / `?focus` / back-nav seed) consumed by `UnravelController` on camera arrival; distance-based auto-collapse remains. Back-nav fan restore keys on the surviving `getDiveContext()` (render-phase read), NOT `useIsReturningFromDocument` ‚Äî the pathname trick reads the already-updated path when a code-split route mounts a pass after the root's location commit. Device-class quality flags live in `canvas/canvasQuality.ts` (coarse pointer ‚Üí dpr 1.5, no AA, demand frameloop after 1.5s, 12-card fan, no date labels); fan columns are viewport-aspect-capped in `unravel.ts fanMaxCols`.
- `2026-07-17` ‚Äî **Structured outputs require the `zod/v4` subpath**, not the default `zod` entry. `@anthropic-ai/sdk` 0.107's `helpers/zod` does `import * as z from 'zod/v4'`, so a schema built with the repo-default `import { z } from 'zod'` (v3 API, shipped alongside v4 in zod 3.25.76) is a type-mismatch against `zodOutputFormat` and makes `.parse()` infer `parsed_output: {}`. Fix: build any schema passed to `zodOutputFormat` with `import * as z from 'zod/v4'` (see `apps/backend/src/llm/llm-response.schema.ts`). Keep such schemas backend-local ‚Äî they're model contracts, not the v3 `@reverie/shared` API contracts.
